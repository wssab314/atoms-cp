# Gemini R5.2｜发布中心与版本历史体验打磨迭代要求

## 背景
Codex 已复验 Gemini R5.1，结论为：R5.1 可接受，无阻塞问题。当前前端版本页、发布页、回退弹窗、GitHub handoff 计划态脱敏均已通过单测、构建和浏览器实测。

本轮 R5.2 是小步打磨，不新增后端能力，不改变 API 契约，目标是把普通用户看到的版本历史与发布中心从“功能可用”推进到“内测可交付”。

## 本轮目标
- 发布页所有阻断原因必须产品化中文展示，不能直接透出后端英文或半技术表达。
- 版本历史详情必须适配真实 `ProjectVersionRecord`，不再只依赖 demo v1/v2/v3 的固定变更文案。
- 发布页补齐无快照、构建失败、回退构建中、GitHub handoff 失败等状态。
- 复查 `/app/:projectId/versions` 与 `/app/:projectId/publish` 在移动端的布局，不出现按钮挤压、横向溢出、文字竖排。

## Codex 复验发现
R5.1 通过以下复验：

- `pnpm --filter @atoms-cp/web typecheck`
- `pnpm --filter @atoms-cp/web test`
- `pnpm --filter @atoms-cp/web build`
- `pnpm -r typecheck`
- `pnpm -r test`
- 浏览器实测 `/app/demo/versions`、`/app/demo/publish`

非阻塞但需要 R5.2 修复的问题：

- Publish 页直接渲染后端 `blockingReasons`，后端可能返回英文提示，例如 `Create an active ready preview snapshot before release.`。普通用户页不能出现这种英文/半技术表达。
- Versions 页真实版本的“变更清单”区域仍偏 demo 固定内容，真实版本没有对应 v1/v2/v3 时信息不足。
- Publish 页对无 active ready snapshot、构建失败、GitHub handoff 失败、保存部署 URL 失败等状态还可以更清楚。
- Versions / Publish 页面需要移动端复查，尤其是按钮组、确认弹窗、发布检查列表、GitHub handoff 面板。

## 具体要求

### 1. Publish blockingReasons 中文化
新增前端本地映射函数，例如 `formatPublishBlockingReason(reason: string): string`。

至少覆盖以下后端或历史提示：

- `Create an active ready preview snapshot before release.` -> `请等待预览快照准备完成`
- `Generate files and run a successful cloud preview build first.` -> `请先完成一次成功的预览构建`
- `Connect GitHub before committing project files.` -> `请先连接代码托管账号`
- `Confirm VITE_SUPABASE_URL and VITE_SUPABASE_ANON_KEY in the deploy target before marking deployed.` -> `请先确认发布环境中的数据库公开配置`
- `Run a passing Supabase live connection test before release.` -> `请先完成数据库连接测试`

未知 reason 也必须做兜底处理：

- 去除 `status code`、`stack`、`workspace`、`Docker`、`Vite`、`pnpm`、`stdout`、`stderr` 等技术词。
- 如果无法安全转译，显示 `当前发布条件尚未满足，请按左侧检查项完成配置`。

### 2. Versions 真实变更清单
真实版本详情区域不能只按 `currentVersion.id === 'v1'/'v2'/'v3'` 渲染。

对真实版本至少展示：

- 版本来源中文：初始生成、手动微调、AI 微调、代码修改、版本回退、发布记录。
- 版本摘要：优先使用 `summary`。
- 变更数量：根据 `changedFiles.length` 展示 `包含 N 项文件变更`，但禁止展示具体文件路径。
- 父版本关系：若存在 `parentVersionId`，展示 `基于上一稳定版本生成` 或类似产品文案，不展示 ID 原文。
- 若快照 failed，展示友好提示：`该版本预览生成失败，可稍后重新生成或回退到稳定版本`。

Demo v1/v2/v3 的丰富文案可以保留，但真实数据必须有通用 fallback。

### 3. Publish 状态补齐
发布确认卡需要区分：

- 无 ready active 快照：主按钮禁用，提示 `请等待预览快照准备完成`。
- 最新构建失败：提示 `最近一次预览生成失败，请先修复后再发布`。
- GitHub handoff 计划失败：显示 `暂时无法准备代码托管，请稍后重试`。
- GitHub handoff confirm 失败：显示 `代码托管推送失败，请稍后重试`。
- 部署 URL 409：保持 R5.1 文案 `该域名已被其他项目占用，请换个网址重试`。

错误文案必须对普通用户友好，不能出现 HTTP code、stack trace、绝对路径或底层工具词。

### 4. 移动端布局复查
重点检查 390px 视口：

- Versions 页右侧按钮组在窄屏下可换行，不被裁切。
- 回退确认弹窗宽度适配 390px，按钮不溢出。
- Publish 页 GitHub handoff 面板在窄屏下不横向溢出。
- 发布检查列表状态 badge 不挤压正文。
- 主发布按钮文字不换成竖排。

如果需要新增 CSS，只做局部响应式规则，不重写整体视觉系统。

## 禁用词与脱敏
普通用户页面继续禁止出现：

- Codex
- Docker
- Vite
- pnpm
- terminal
- stdout
- stderr
- workspace
- dist
- node_modules
- stack
- status code

允许在 `api.ts` 类型定义、测试 mock、Admin 页面中出现技术字段；但普通用户可见 DOM 不能出现。
