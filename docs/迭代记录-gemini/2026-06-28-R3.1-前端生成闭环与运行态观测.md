# Gemini R3.1｜前端生成闭环与运行态观测迭代要求

## 背景

Codex 已完成 R3 后端/Agent Worker 运行闭环：

- 新增 `codex-worker` 运行态入口，支持 `dry_run` 与 `docker` fixture mode。
- CodexTask 状态闭环为 `queued -> claimed -> preparing_workspace -> codex_running -> validating -> succeeded/failed`。
- builder-worker 会在 build 开始、preview snapshot 创建、build 完成时写入 trace。
- Preview route 已补 `cache-control`、CSP、iframe/security headers。
- 同一 project 同时只允许一个 active writer。

Gemini R2.2 已修复最新任务/快照排序、React `act(...)` warning，并通过 `pnpm --filter @atoms-cp/web test`。但 Codex 复验发现仍有一个 P0 残留：

- `/dashboard` 新建项目后，`generateProjectAppSpec -> generateDesignProfiles -> createCodexTask` 任一步失败时，目前只是 `console.warn`，仍会跳转真实项目 `/app/:projectId/generating`。
- 这会造成“看起来开始生成，但实际没有 CodexTask 入队”的假运行态。
- 本轮必须先修这个链路，再做 R3 运行态观测。

## 目标

在不改后端 API、不新增依赖的前提下，让前端从“能读 API”升级为“能正确表达真实生成闭环”：

- Dashboard 启动生成必须保证真实任务已入队，失败时不得进入假真实生成页。
- Generating 页必须等到 preview snapshot ready 后再进入工作台，而不是 CodexTask 一 succeeded 就跳转。
- Workbench / Versions / Publish 必须只把真实 ready snapshot 当作可预览版本。
- Admin 可以显示 R3 worker/build/trace 运行态，普通用户页面继续屏蔽底层技术词。

## 实现范围

只修改：

- `apps/web/src/App.tsx`
- `apps/web/src/styles.css`
- `apps/web/src/App.test.tsx`
- 如确有必要，可最小修改 `apps/web/src/api.ts` 类型定义。

不要修改：

- `apps/api/**`
- `apps/builder-worker/**`
- `packages/**`
- migration / docker compose

## 必做项

### 1. 修复 Dashboard 真实生成链路

将 `/dashboard` 的开始生成流程改成严格串行 pipeline：

1. `createProject(payload)`
2. `generateProjectAppSpec(projectId)`
3. `generateDesignProfiles(projectId)`
4. `createCodexTask(projectId)`
5. 只有第 4 步成功后，才跳转 `/app/:projectId/generating`

要求：

- 任一步失败，都必须进入统一失败处理，不得吞掉错误后继续跳真实项目 generating。
- 用户可见文案使用“生成准备失败，请稍后重试”或同等产品化表达。
- 可以保留 mock fallback，但 fallback 必须使用 `mock-*` id，不能让真实项目进入无任务 generating 页。
- 不在普通用户页面显示 `AppSpec`、`DesignProfile`、`CodexTask`、`worker` 等技术词。

### 2. 修复 Generating 页完成条件

当前逻辑不能只看 latest task `succeeded` 就跳转工作台，因为 R3 中 task succeeded 只代表已创建 build job，preview snapshot 可能还没 ready。

新的完成条件：

- 轮询 `fetchCodexTasks(projectId)`。
- 同时轮询 `fetchPreviewSnapshots(projectId)`。
- task `failed`：停止轮询，显示用户化失败状态。
- task `succeeded` 但没有 ready snapshot：停留在“正在准备预览快照”阶段。
- 存在 `status === "ready"` 且 `active === true` 的 snapshot：跳转 `/app/:projectId`。
- 如果没有 active ready snapshot，但有最新 ready snapshot，可作为兜底跳转。

要求：

- 轮询间隔保持 3 秒。
- 页面卸载必须清理 timer。
- 不出现空白页、无限 loading 或不可解释状态。
- 错误摘要需要产品化，不直接展示后端英文堆栈或技术词。

### 3. Workbench 真实快照展示

`/app/:projectId` 的右侧应用查看器：

- 对真实 project，只允许使用 ready snapshot 的 `url` 作为 iframe src。
- 如果真实 project 尚无 ready snapshot，显示空态：“预览快照正在准备中”，不要拼 `/preview/${projectId}/v3` 这种假 URL。
- demo/mock project 可以继续使用 fixture。
- snapshot 选择优先级：active ready snapshot > latest ready snapshot > 空态。
- 左侧构建记录状态要能表达：
  - 等待开始
  - 正在准备
  - 正在生成应用
  - 正在校验结果
  - 正在准备预览快照
  - 预览已就绪
  - 生成失败

### 4. Versions / Publish 快照口径统一

- `/app/:projectId/versions` 只把 ready snapshots 显示为可回退/可发布版本。
- failed / creating snapshots 可以显示为历史事件，但不能显示“一键发布”或“可回退”主操作。
- `/app/:projectId/publish` 的发布确认必须绑定 active ready snapshot；没有 ready snapshot 时禁用确认按钮并提示“请等待预览快照准备完成”。

### 5. Admin 运行态观测增强

Admin 页面允许出现技术词，但必须保持简洁、脱敏。

要求：

- 使用已有 `fetchAdminOperations()` 数据，不新增后端 API。
- 展示最近 Codex tasks 的状态、attemptCount、projectId、createdAt。
- 展示最近 preview snapshots 的 status、active、projectVersionId。
- 展示最近 trace events 的 type、message、createdAt。
- secret / token / key 字段继续强制掩码。
- 不展示原始用户 prompt、密钥、完整环境变量、宿主路径。

### 6. 普通用户技术词屏蔽

普通用户页面继续禁止出现以下英文技术词：

- `Codex`
- `Docker`
- `Vite`
- `pnpm`
- `dist`
- `HMR`
- `WebContainer`
- `terminal`
- `stdout`
- `stderr`
- `workspace`
- `AppSpec`
- `DesignProfile`
- `build job`
- `worker`

Admin 页面例外。

## 测试要求

在 `App.test.tsx` 增加或修正测试，至少覆盖：

- Dashboard 成功链路会依次调用 `createProject -> generateProjectAppSpec -> generateDesignProfiles -> createCodexTask`。
- `generateProjectAppSpec` 失败时，不调用 `createCodexTask`，不进入真实项目 generating。
- `generateDesignProfiles` 失败时，不调用 `createCodexTask`，进入 mock fallback 或显示失败状态。
- `createCodexTask` 失败时，不进入真实项目 generating。
- Generating 页在 task `succeeded` 但 snapshot 未 ready 时不会跳转工作台。
- Generating 页在 active ready snapshot 出现后跳转工作台。
- Workbench 对真实 project 无 ready snapshot 时显示空态，不拼假 preview URL。
- ready snapshot 排序与 active 优先级正确。
- 普通用户页不出现禁用技术词。
- Admin secret masking 仍通过。
- 测试运行时不得出现 React `act(...)` warning。

Mock API 即可，不要真实请求后端。

## 实际完成情况（Gemini 2026-06-28）

- **修复摘要**：修复了 R2.2 残留的假生成链路逻辑；重构了双重条件 Generating 页轮询；禁止了真实项目在无快照时使用假 URL 预览；限制了版本的操作和发布只允许在 ready 快照状态下进行；丰富了 Admin 后台观测指标。
