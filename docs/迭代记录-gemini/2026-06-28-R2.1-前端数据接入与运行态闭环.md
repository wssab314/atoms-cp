# Gemini R2.1｜前端数据接入与运行态闭环迭代要求

## 当前项目状态

Codex 复核后判断：项目已经从“旧 deterministic codegen MVP”推进到“新架构可施工骨架”阶段。

当前已完成：

- 前端 R1/R1.2：Dashboard Shell、Editor Shell、路由、Quiet App Builder 视觉系统、移动端顶栏返修均已通过 Codex Browser QA。
- 后端 R2：`WorkspaceRecord`、`CodexTaskRecord`、`PreviewSnapshotRecord`、`TraceEventRecord` schema 已落地。
- 后端 R2：`workspaces`、`codex_tasks`、`preview_snapshots`、`trace_events` migration 与 store 已落地。
- 后端 R2：CodexTask Planner 已能从 AppSpec + DesignProfile 生成结构化 `taskSpec`，且不直接传原始用户 prompt。
- 后端 R2：dry-run Codex worker 已能领取 queued task、锁定 workspace、生成受控 Vite workspace、写入 project version / project files、创建 build job、记录 trace。
- 后端 R2：builder-worker 已能在成功 build 后写入 `preview_snapshots`，新 preview URL 主路径使用 snapshot id。

当前主要缺口：

- 前端仍大量依赖 `App.tsx` 内静态 fixture，尚未把 Shell 接入真实后端状态。
- Editor 的“构建记录 / 预览快照 / 版本状态 / trace 时间线”仍是 mock，不反映真实 `codex_tasks`、`workspaces`、`preview_snapshots`、`trace_events`。
- Dashboard / Projects 的创建与进入项目流程仍更像演示稿，尚未形成真实项目入口。
- Codex 侧下一轮将继续推进真实 Docker Codex Worker 与 worker 触发链路；Gemini 本轮只处理前端 API 接入和运行态表现。

整体进度估算：

- P0 可演示闭环：约 **42%**。
- 前端 Shell 与视觉系统：约 **75%**。
- 后端数据底座与 Agent 执行骨架：约 **50%**。
- 真实 Docker Codex Worker / 生产级 build-preview-deploy 闭环：约 **25%**。

## 本轮目标

把 R1.2 前端 Shell 从“纯静态高保真 mock”推进到“可消费后端真实状态的运行态 UI”。

本轮不是重新设计页面，也不是改后端。目标是保留 R1.2 视觉与响应式成果，在 `apps/web` 内建立真实 API 数据接入、加载态、错误态、空态与 fixture fallback，让下一轮 Codex 后端接入真实 Docker Worker 时，前端已经具备承接能力。

## API 接入范围

优先使用 `apps/web/src/api.ts` 已存在函数：

- `fetchProjects()`
- `createProject(payload)`
- `generateAppSpec(projectId)`
- `generateDesignProfiles(projectId)`
- `selectDesignProfile(projectId, designId)`
- `createCodexTask(projectId)`
- `fetchCodexTasks(projectId)`
- `fetchProjectWorkspaces(projectId)`
- `fetchPreviewSnapshots(projectId)`
- `fetchTraceEvents(projectId, limit)`
- `fetchProjectPublishState(projectId)`

本轮可以保留 fixture fallback：

- 当 API 返回 404、409、空数组或本地后端未启动时，页面仍应展示 R1.2 的静态演示数据。
- 但如果 API 成功返回真实数据，页面必须优先渲染真实数据。
- 不要让用户看到 `404`、`409`、`stack trace`、`fetch failed` 等工程错误文本；普通用户页使用产品化错误文案。

## 页面任务

### 1. `/projects` 接入真实项目列表

要求：

- 页面加载时调用 `fetchProjects()`。
- 成功时用真实项目卡片替换或合并现有 fixture 项目。
- 空项目时展示产品化空态，并提供“新建应用”入口。
- 请求失败时展示非阻塞错误提示，同时保留 fixture 示例卡片作为演示 fallback。
- 搜索、状态筛选、排序、视图切换继续可用。

普通用户文案：

- 使用“草稿”“方案已生成”“正在构建”“可预览”“已发布”等用户语义。
- 不出现 `Codex`、`workspace`、`Vite`、`pnpm`、`build job`、`trace`、`stdout`、`stderr` 等技术词。

### 2. `/dashboard` Prompt Composer 创建真实项目

要求：

- 用户点击“开始生成”时：
  1. 若 Prompt 为空，显示本地校验提示。
  2. 若 Prompt 有内容，调用 `createProject({ name, prompt, target })`。
  3. 创建成功后进入一个生成准备态，可以导航到 `/projects` 或 `/app/:projectId/generating`。
- 可以先不自动调用 Spec/Design/CodexTask 全链路，但按钮状态必须真实：
  - idle
  - submitting
  - success
  - error
- 示例卡片填充 Prompt 的交互必须保留。

建议：

- 项目名可从 Prompt 前 12-18 个中文字符生成，或使用“新应用草稿”。
- 错误文案应是“创建失败，请稍后重试”，不要展示底层异常。

### 3. `/app/:projectId` 接入 CodexTask / Workspace / PreviewSnapshot / Trace

要求：

- 页面进入时并发请求：
  - `fetchCodexTasks(projectId)`
  - `fetchProjectWorkspaces(projectId)`
  - `fetchPreviewSnapshots(projectId)`
  - `fetchTraceEvents(projectId, 20)`
- 如果真实数据存在：
  - 左侧“构建记录”显示最近 CodexTask 的用户化状态；
  - 右侧“应用查看器”优先展示 active/latest `previewSnapshot.url`；
  - 没有 preview snapshot 时，继续使用现有 mock preview；
  - trace events 用于补充构建记录时间线，但普通用户页不要直接显示技术事件名。
- 如果真实数据为空或请求失败：
  - 保留 R1.2 fixture 演示，不让页面空白。

状态映射要求：

- `queued` -> `等待开始`
- `claimed` -> `已进入队列`
- `preparing_workspace` -> `正在准备项目空间`
- `codex_running` -> `正在生成应用`
- `validating` -> `正在校验结果`
- `running` -> `正在处理`
- `succeeded` -> `已生成预览`
- `failed` -> `生成失败`
- `cancelled` -> `已取消`

普通用户页限制：

- 不显示 `Codex`。
- 不显示 `workspace`。
- 不显示 `taskSpec`。
- 不显示本地路径。
- 不显示 `stdout`、`stderr`。
- 不显示 `Docker`、`Vite`、`pnpm`、`dist`、`HMR`、`WebContainer`、`terminal`。

### 4. `/app/:projectId/generating` 变成真实任务进度页

要求：

- 优先读取最近未完成的 CodexTask。
- 根据任务状态高亮对应步骤。
- 如果没有真实任务，继续展示 R1.2 mock 生成时间线。
- 页面文案仍使用用户语义，例如“正在生成应用”“正在准备预览快照”。

### 5. `/app/:projectId/versions` 接入 preview snapshots

要求：

- 调用 `fetchPreviewSnapshots(projectId)`。
- 如果有真实 snapshots，版本列表应优先来自 snapshots。
- active snapshot 显示“当前稳定版本”。
- failed snapshot 显示“生成失败”，并展示用户化错误摘要。
- 空状态使用 fixture fallback。

### 6. `/app/:projectId/publish` 接入 publish state

要求：

- 调用 `fetchProjectPublishState(projectId)`。
- 成功时用真实 GitHub / Supabase / Vercel readiness 渲染发布检查项。
- 失败时保留 R1.2 fixture 发布检查项。
- 不改变发布按钮的现有 mock 流程，本轮不触发真实部署。

### 7. `/admin` 可显示技术状态，但保持安全

要求：

- Admin 页面允许显示 `CodexTask`、`workspace`、`preview snapshot`、`trace event` 等技术词。
- secret 仍必须 masked。
- 不显示明文 token、service role key、API key。
- 如果接入 admin overview 现有 API，必须保留 loading/error/empty 状态。

## 设计与 UX 要求

- 保持 R1.2 Quiet App Builder 视觉系统，不做大面积重绘。
- 不改变 Dashboard Shell 和 Editor Shell 的信息架构。
- 不破坏 390px 移动端顶栏修复。
- 所有 API loading 必须使用轻量 skeleton / shimmer / subdued spinner，不使用终端风或工程日志风。
- 普通用户页所有错误态都使用产品化文案。
- 可见按钮仍使用 lucide-react 图标。
