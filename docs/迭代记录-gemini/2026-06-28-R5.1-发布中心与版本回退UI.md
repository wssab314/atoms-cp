# Gemini R5.1｜发布中心与版本回退 UI 迭代要求

## 背景
Codex R6 已补齐发布闭环后端能力：

- `POST /api/projects/:projectId/versions/:versionId/rollback`
- `POST /api/projects/:projectId/preview-snapshots/:snapshotId/activate`
- `GET /api/projects/:projectId/publish` 已返回 `currentVersionId`、`activePreviewSnapshotId`、`canPublish`、`blockingReasons`
- 保存部署 URL 前，后端要求项目已有 active ready 预览快照

本轮 Gemini 只处理 `apps/web` 前端 UI/UX 和测试，不改后端、shared schema、migration、worker、builder-worker。

## 目标
把 `/app/:projectId/versions` 与 `/app/:projectId/publish` 从展示型页面升级为真实可操作闭环：

- 用户可以查看真实版本历史，选择 ready 快照设为当前预览。
- 用户可以对历史版本发起“回退”，确认后进入构建等待。
- 发布中心根据真实后端 `canPublish` 和 `blockingReasons` 决定按钮状态。
- 用户可以完成 GitHub handoff、查看发布检查项、保存部署 URL。

## 新接口契约

### 版本回退
`POST /api/projects/:projectId/versions/:versionId/rollback`

返回：

```ts
{
  projectVersion: ProjectVersionRecord;
  buildJob: BuildJobRecord;
  traceEvent: TraceEventRecord;
}
```

前端必须在二次确认后调用。成功后跳转 `/app/:projectId/generating`，等待新版本生成新的 ready 预览快照。

### 激活预览快照
`POST /api/projects/:projectId/preview-snapshots/:snapshotId/activate`

返回：

```ts
{
  previewSnapshot: PreviewSnapshotRecord;
  traceEvent: TraceEventRecord;
}
```

只有 `status === 'ready'` 的快照允许激活。非 ready 快照按钮禁用，并显示“快照未就绪，无法设为当前版本”。

### 发布状态
`GET /api/projects/:projectId/publish`

新增字段：

```ts
{
  currentVersionId?: string;
  activePreviewSnapshotId?: string;
  canPublish: boolean;
  blockingReasons: string[];
}
```

前端必须直接信任 `canPublish` 控制发布主按钮，不要自行拼接发布条件。

## 实现范围

### 1. 扩展 API 使用
`apps/web/src/api.ts` 已提供：

- `rollbackProjectVersion(projectId, versionId)`
- `activatePreviewSnapshot(projectId, snapshotId)`
- `fetchProjectPublishState(projectId)`
- `fetchProjectVersions(projectId)`
- `fetchPreviewSnapshots(projectId)`

如发现类型缺口，只补 API client 类型，不改后端返回结构。

### 2. Versions 页面
`/app/:projectId/versions` 使用真实 versions + snapshots：

- 版本时间线按 `version` 倒序。
- source 文案映射：
  - `initial_generate` -> 初始生成
  - `selector_edit` -> 手动微调
  - `agent_patch` -> AI 微调
  - `code_edit` -> 代码修改
  - `rollback` -> 版本回退
  - `deploy` -> 发布记录
- 当前 active snapshot 所属版本展示“当前预览”。
- ready snapshot 展示“设为当前版本”。
- 非 ready snapshot 展示“快照未就绪，无法预览”。
- 禁止展示 `workspacePath`、文件路径、内部 source 原文。

### 3. 回退交互
对历史版本提供“回退到此版本”：

- 当前版本不显示回退按钮。
- 点击后必须二次确认，确认卡文案使用“将基于该稳定版本重新生成一个新版本”。
- 调用 rollback 成功后显示“已开始恢复稳定版本”，并跳转 generating。
- 失败时展示友好错误，不出现 status code、stack、workspace 等技术词。

### 4. Publish 页面
`/app/:projectId/publish` 接入真实 `ProjectPublishState`：

- 检查项直接使用 `checklist`。
- `blockingReasons` 用产品化提示展示在确认卡上。
- `canPublish === false` 时禁用发布主按钮。
- `activePreviewSnapshotId` 缺失时显示“请等待预览快照准备完成”。
- 保存部署 URL 使用现有 `saveProjectDeploymentUrl`，后端 409 时转成友好提示。

### 5. GitHub handoff
沿用现有 GitHub 连接和 commit flow：

- 计划态只展示文件数量、目标仓库、分支、提交说明。
- 禁止展示文件内容、文件系统路径、workspace、hash 原文。
- commit 成功后刷新 publish state，展示“代码已提交”。

## 禁用词与脱敏
普通用户页面继续禁止出现：

- Codex
- Docker
- Vite
- pnpm
- terminal
- stdout
- stderr
- workspace
- dist
- node_modules

Admin 页面可以显示技术状态，但仍不能泄漏 secret、token、绝对路径。
